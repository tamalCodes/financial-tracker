# DATA_MODEL — next-ver

Schema **inferred from Supabase queries** (no migration files in repo). Fields marked
_(inferred)_ are not directly proven by a SELECT/INSERT and are best-guess; verify against
the live Supabase project before relying on them. All tables are owned per-user via `user_id`.

## Ownership rule
Every row belongs to a user. Every query filters `.eq("user_id", userId)`. No reliance on
Postgres RLS is visible in code — ownership is enforced in the query layer.

## Month key
A "month" is the string `YYYY-MM-01` (day always `01`). Generated by `formatMonthKey`,
shifted by `shiftMonthKey(key, ±1)`. Stored in `credits.month`, `expenses.month`,
`investments.start_month`, `monthly_balances.month`.

## Tables

### `credits` — income entries
| column | type | notes |
|--------|------|-------|
| `id` | uuid | PK _(inferred)_ |
| `user_id` | uuid | owner; FK → auth.users _(inferred)_ |
| `month` | text | `YYYY-MM-01` |
| `description` | text | |
| `amount` | numeric | positive income |
| `carry_forward` | bool | if true, copied into next month on load |
| `carried_from_month` | text\|null | set when auto-carried; source month |
| `created_at` | timestamptz | default now() _(inferred)_; ordered desc |

### `expenses` — spending entries
Same shape as `credits` (`id, user_id, month, description, amount, carry_forward,
carried_from_month, created_at`). `amount` is a positive cost (subtracted from balance).

### `investments` — recurring/active investments
| column | type | notes |
|--------|------|-------|
| `id` | uuid | PK _(inferred)_ |
| `user_id` | uuid | owner |
| `start_month` | text | `YYYY-MM-01`; first active month |
| `description` | text | |
| `amount` | numeric | positive; subtracted from balance |
| `is_active` | bool | soft-delete flag (DELETE sets `false`) |
| `carry_forward` | bool\|null | true ⇒ counts in every month ≥ `start_month`; else only its `start_month` |
| `created_at` | timestamptz | _(inferred)_; ordered desc |

Read filter: `lte("start_month", currentMonth)` + `is_active = true`, then in code keep rows
where `carry_forward` is true OR `start_month === currentMonth`.

### `monthly_balances` — per-user, per-month balance
| column | type | notes |
|--------|------|-------|
| `id` | uuid | PK _(inferred)_ |
| `user_id` | uuid | owner |
| `month` | text | `YYYY-MM-01`; unique with `user_id` (insert handles `23505`) |
| `starting_balance` | numeric | opening balance for the month |
| `closing_balance` | numeric | maintained incrementally (see invariant) |

`(user_id, month)` is effectively unique — `loadDashboardData` catches Postgres `23505`
(unique violation) on concurrent insert.

## Carry-forward semantics
- **Credits / expenses**: on `loadDashboardData(currentMonth)`, rows in `previousMonth` with
  `carry_forward = true` are **copied** into `currentMonth` (new rows, `carried_from_month =
  previousMonth`), deduped by `description|amount` so reloads don't duplicate. See
  `ensureCarryForwardCredits` / `ensureCarryForwardExpenses` in `lib/api/dashboard.ts`.
- **Investments**: NOT copied. A single row counts in `currentMonth` if `carry_forward` is
  true (any month ≥ `start_month`) or its `start_month` equals `currentMonth`.

## Balances invariant
```
closing_balance = starting_balance + Σcredits − Σexpenses − Σinvestments   (for the month)
```
- `calculateClosingBalance(starting, credits, expenses, investments)` computes it.
- Maintained **incrementally** by `applyBalanceDelta(supabase, userId, month, delta)` on each
  mutation (delta sign per CONVENTIONS §4) — no full recompute on the hot path.
- `loadDashboardData` **self-heals**: if stored `closing_balance` ≠ recomputed, it rewrites
  via `updateClosingBalance`.
- **Month rollover**: when a month has no balance row, `loadDashboardData` seeds it from the
  previous month's `closing_balance` (becomes both starting and closing).
- Editing the starting balance (`/api/balances` PUT) recomputes closing from the full lists
  via `updateClosingBalance(..., { updateStarting: true })`.
